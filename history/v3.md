### JHP 3.9.0 (Apr 2026)

- **`includeSearchRoots` on `JHP#process()`.** Optional array of **absolute** directory paths. When set (and `includePathResolver` is not), the built-in `$include` logic tries the including file’s directory first, then each root in order (similar to a PHP `include_path` style list). For paths that start with `/`, each root is tried with the path after the leading `/`. If the include string starts with `../` (or is `..`), resolution uses **only** the including file’s directory, not the search list. Omitted, null, or `[]` leaves the previous 3.8-style single-`#rootDir` rules. `includePathResolver` still wins when both are passed (search roots are ignored for that `process()`). Tests in `test/include.test.js`.
- **Docs.** README “Include paths” section rewritten as resolver, then search roots, then default.

### JHP 3.8.0 (Apr 2026)

- **`includePathResolver` on `JHP#process()`.** Optional `includePathResolver( file, currentDir )` **fully replaces** built-in `$include` path resolution for that `process()` call. Return an absolute (or resolvable) path to an existing file, or `null` for “unresolved” (same `<< Error: Unable to resolve include path '…' >>` output as a missing file). If omitted, behavior matches previous built-in rules. The resolver is cleared after each `process()`; nested includes use the same function with `currentDir` set to the including file’s directory. Host apps can implement `templateDir` / `srcDir` / project-safety policy without `registerDollarProperty( 'include', … )`.
- **Docs and examples.** README “Include paths” section; example `build.js` sets `rootDir` to `examples/src` with root-style `/partials/...` includes in `index.jhp`. Pathing tests in `test/include.test.js` assert the exact unresolved-include error line.

### JHP 3.5.0 (Mar 2026)

- **Test suite added.** Node built-in test runner (`node --test test/*.test.js`). Script: `pnpm test`. Tests cover echo, conditionals, define/context, errors, include, processors, example-site build, and variable-declaration edge cases (strings with semicolons, multiline, cross-file declare/use, comments, unclosed strings).
- **Example site build.** Single `examples/build.js` exports `buildExamples()` (used by tests and by `npm run build` when run directly); demo script removed from package.json.
- **Variable declarations: multiline and semicolon-in-string fix.** Script blocks are now split into logical statements via a state machine that respects strings (`'` `"` `` ` ``), template `${ }`, and braces `{ }`. Statement-ending semicolons inside strings or templates no longer break parsing. Multiline declarations are supported.
- **Comments in script blocks.** State machine tracks `//` and `/* */` so semicolons inside comments are not treated as statement boundaries.
- **Unclosed string/template handling.** When a declaration has an unclosed quote or backtick, JHP outputs `<< Error: Unclosed string or template literal. >>` and sets the variable to `''` (and runs `$.context`) so later template code does not throw.
- **Removed** unused `#regex.declaration`; declaration boundaries are determined only by the state machine.

### JHP 3.2.0 (12 Dec 2025)

- Identified and fixed an issue `createNode` and insert methods where not properly adding closing tags for elements that require them.
- Updated documentation and improved examples for better clarity on usage of new features.

### JHP 3.1.0 (12 Dec 2025)

We have upgraded the Simple HTML Parser to the standalone package `@caboodle-tech/simple-html-parser`. This means that JHP no longer includes its own copy of the
Simple HTML Parser code. This change helps to reduce duplication and ensures that JHP always uses the latest version of the Simple HTML Parser.

### JHP 3.0.0 (15 Oct 2025)

We now include the pre and post processor scripts in the main JHP package. This means you no longer
need to include them separately yourself. By default they will be included automatically when you
get a new instance of JHP. If you do not want this behavior, you can pass
`{ reregisterJhpProcessors: false }` in the JHP constructor options object.
